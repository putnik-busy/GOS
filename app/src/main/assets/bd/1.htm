Реляционная база данных (БД) – это набор взаимосвязанных данных, хранящихся в таблицах вместе с некоторыми программами по их обработке (хранимые процедуры и триггеры).

Таблицы бывают постоянные (Таблицы - Table), временные (Курсоры - Cursor) и виртуальные (Просмотры, Обзоры или Представления - View).

  Таблицы – основное реальное хранилище данных в составе БД. Они состоят из записей (строк), каждая из которых имеет одинаковый вид (структуру), определяемый списком полей (колонок). Набор полей определяет структуру таблицы. Каждое поле имеет уникальное имя, тип (например, числовой, символьный, дата или др.), ширину. На диске таблицы занимают один или несколько файлов. Данные полей с неопределенной длиной обычно хранятся в отдельных вспомогательных файлах, например, поля примечаний, а в самой таблице хранится только указатель на место в этом вспомогательном файле. Имеется набор основных операций для обработки записей таблиц – добавление записей, изменение значений полей, удаление записей, поиск, упорядочивание. Эти операции могут выполняться над отдельными записями или над группами записей.

Курсоры – это временные таблицы, которые обычно получаются в результате запросов к другим таблицам. Они хранятся в оперативной памяти и автоматически уничтожаются при завершении работы программы (приложения), создавшей и использовавшей их. Курсор доступен только тому приложению (программе), которое создало его. К курсорам можно применять все основные операции над таблицами. Курсоры бывают изменяемые (создаются оператором CREATE CURSOR) и неизменяемые (получаются как результат запроса оператором SELECT-SQL).

  Просмотры – это виртуальные таблицы, которые строятся на основе запросов к другим таблицам и обычно используются для просмотра полученной информации. Реально информация хранится в источниках – таблицах и курсорах. Локальные просмотры строятся на основе источников «родного» формата, т.е. на базе таблиц той же СУБД. Если в качестве источников выступают данные других форматов (например, MS SQL Server или Oracle), то такие просмотры называются удаленными и для работы с ними надо устанавливать специальные соединения (Connection) с другими СУБД. В этом случае надо использовать специальные программы-драйверы, преобразующие таблицы и данные из одного формата в другой. В некоторых случаях через просмотры можно менять значения полей в таблицах-источниках, где хранятся данные.

  Для создания постоянной таблицы необходимо задать ее структуру в операторе CREATE TABLE <таблица> (<поле-1>, <поле-2>, …<поле-N>), т.е. указать имя таблицы с местоположением на диске и описать вид каждого поля. При этом можно указывать значение поля по умолчанию, возможность пустого значения, наличие простых индексов по полям и условия достоверности данных на уровне поля и на уровне записи. Это набор логических условий, которым должны удовлетворять новые данные, которые добавляются или заменяют старые значения в полях.

Таблицы, как и любые файлы, перед использованием должны быть подготовлены к работе (открыты), а после окончания обработки желательно их закрывать. Для открытия используется оператор

USE <таблица> ALIAS <псевдоним> <режим обработки> <порядок>

Для закрытия таблицы используется тот же оператор в другой форме

USE <таблица>

Между таблицами могут быть установлены постоянные отношения или временные связи. Отношения используются для автоматического поддержания ссылочной целостности данных во взаимосвязанных таблицах, а временные связи между ними устанавливаются только на время выполнения какого-то фрагмента программы. В связанных таблицах одна из таблиц (на которую ссылаются из другой таблицы) называется родительской, другая – дочерней (таблица, которая ссылается на другую). Родительская таблица обязательно должна быть проиндексирована, так как именно за счет индексов достигается быстрая связь между записями таблиц. Для постоянных отношений индекс родительской таблицы должен быть первичным, т.е. с уникальными значениями.

Для обработки таблиц в сетевых многопользовательских приложениях могут использоваться два режима – монопольный (доступ только для одного приложения) и разделяемый (одновременный доступ для нескольких приложений). Режим обработки указывается при открытии таблицы, т.е. при подготовке ее к работе.

USE <таблица> IN 0 ALIAS <псевдоним> EXCLUSIVE | SHARED

В первом случае (EXCLUSIVE) таблица может использоваться только одним приложением, и допустимы любые операции над ней, даже изменяющие ее структуру. В разделяемом режиме (SHARED) просматривать записи могут одновременно несколько приложений, но вносить изменения в каждый момент времени может только одно из них. Некоторые операции в этом случае вообще недопустимы, например, создание индексов, упаковка, изменение структуры таблицы и другие.

Для временного монопольного захвата отдельных записей (например, с целью внесения в них изменений) используются блокировки. Если некоторое приложение заблокировало запись, то для остальных в это время она доступна только для чтения (просмотра). Функция RLOCK () позволяет заблокировать текущую запись в активной таблице, если она не заблокирована другими. Функция FLOCK () позволяет заблокировать всю таблицу, если ни одна запись не заблокирована другими приложениями.

После обработки записей их нужно разблокировать, например, оператором UNLOCK.

Попытка изменить (удалить) запись, заблокированную в данный момент другим приложением, приведет к ошибочной ситуации.

Для того, чтобы данные во взаимосвязанных таблицах были согласованными, т.е. удовлетворяли определенным условиям, используются специальные программы (процедуры и функции). Они обычно хранятся в самой БД, а не приложении, чтобы разные приложения могли одинаково ими пользоваться. Такие программы называются хранимыми процедурами.
